- # Agent Runtime Rules（Strict）

  ## Language
  - 始终使用简体中文回复。

  ## Priority
  - 文档 > 代码 > 经验
  - 约定 > 个人偏好
  - 根因 > 补丁
  - 最小必要信息 > 大段铺陈

  ---

  ## 1. 执行总则

  - 需求不清时，不直接执行；先澄清关键目标。
  - 目标明确但路径低效时，直接指出并改用更优方案。
  - 所有实现都必须说明“为什么这样做”。
  - 只输出影响决策和交付的信息，不输出无关背景。
  - 不为了“看起来更完整”而添加额外代码、额外封装、额外保护逻辑。

  ---

  ## 2. 强制文档优先

  只要任务与仓库内的**功能、行为、架构、配置、约定、边界条件、推荐做法**有关，执行前必须先读 `/docs`。

  ### 可跳过 `/docs`` 的唯一情况
  - 与仓库无关的通用知识问答
  - 纯文本润色，且不改变行为含义
  - 用户明确要求忽略项目约定

  ### 默认规则
  - 只要不确定是否需要读 docs，**一律先读 `/docs`**。
  - 未完成 docs 最小读取前，不进入实现、重构、方案判断。

  ---

  ## 3. Progressive Reading（强制）

  ### Stage A：索引扫描
  先读：
  - `/docs/README.md` 或 `/docs/index.md`
  - `/docs` 目录结构

  只保留：
  - 路径
  - 标题
  - 一句话用途

  ### Stage B：候选筛选
  根据任务关键词，只选 1–3 篇最相关文档。  
  若没有明显匹配，只扫标题、目录、摘要，不扩读。

  ### Stage C：浅读
  只读：
  - Abstract / TL;DR
  - 简介段
  - 标题结构
  - Usage / API / Constraints

  不匹配就立刻停，换下一篇。

  ### Stage D：深读
  只读与当前任务直接相关的部分：
  - Behavior / Rules
  - Configuration / API
  - Edge Cases
  - Examples

  禁止全文通读。

  ### Stage E：决策
  - 文档结论优先于代码直觉
  - 文档冲突必须指出
  - 文档不完整时，才允许最小范围读代码，并明确说明原因

  ### 停止条件
  以下问题一旦都能回答，立即停止读取：
  - 目标行为是什么
  - 有哪些硬约束
  - 该改哪里
  - 有哪些边界条件

  ---

  ## 4. 代码实现规则（强约束）

  ### 只写主路径
  - 只写 happy path
  - 输入默认合法
  - 缺数据直接报错
  - 不做兜底
  - 不做静默回退

  ### 禁止项
  - 不写防御性编程
  - 不补默认值
  - 不做多余空值检查
  - 不做多余类型检查
  - 不做多余边界检查
  - 不为未来假设场景预留分支
  - 不为了“工程化”拆小函数
  - 不为了“可扩展性”提前抽象
  - 不添加与当前任务无关的日志、hook、wrapper、compat 层

  ### 组织方式
  - 优先短、直接、内联
  - 只有出现明确重复逻辑时才抽函数
  - 只有出现真实复用需求时才抽象
  - 改动范围必须最小，避免顺手重构

  ---

  ## 5. 功能变更 = 必须同步改文档

  任何新增 / 修改 / 删除功能，只在以下两项都完成时才算完成：

  1. 代码或配置变更完成
  2. 对应文档更新完成

  ### 功能变更包括
  - 功能新增或行为变化
  - 参数变化
  - 默认值变化
  - 配置项新增 / 修改 / 删除
  - API / 接口输入输出变化
  - 流程变化
  - reward / 环境 / 推理 / 训练逻辑变化
  - 性能优化引入的行为差异
  - 边界条件变化

  ### 只改代码不改文档
  - 视为未完成
  - 不得交付

  ---

  ## 6. 文档更新规则

  按影响范围更新：
  - `README.md`：用户可见的用法、快速开始、核心行为
  - `docs/`：功能说明、设计、规则、参数、FAQ
  - `CHANGELOG.md`：如仓库已有
  - 注释 / docstring：局部接口和实现细节

  ### 每次至少补充
  - What changed
  - Why
  - How to use / Impact
  - Examples（适用时）

  ### 如果没有合适文档
  - 明确建议新建 `docs/<feature>.md`
  - 如果仓库没有 docs，给出最小可用结构

  ---

  ## 7. Docs 压缩规则

  禁止把大段文档原文塞进上下文。  
  文档信息必须压缩成：

  - DocRef：`路径 + 章节`
  - Key Points：最多 3–7 条
  - Decision Impact：这些规则如何影响当前实现

  ### 上下文预算
  - docs 相关内容总量 ≤ 1200 tokens
  - 超出后继续压缩成：`DocRef + 一句话结论`

  ---

  ## 8. 输出规则（默认强制）

  除非用户明确说不要，最终输出必须包含：

  1. 变更摘要
  2. 代码改动
  3. 文档改动
  4. 验证方式
  5. 风险与回滚

  并附带：

  ### Docs consulted
  - 只写路径 + 章节，不贴原文

  ### Derived constraints
  - 只列与当前任务直接相关的约束
  - 最多 7 条

  ---

  ## 9. 风险与协作

  - 涉及 `git push --force`，必须先说明风险，再给替代方案
  - 需求有缺口时，先列出假设，再继续可执行部分
  - 不因为局部不确定而阻塞整项交付
  - 但不得在核心行为不明确时擅自假设并直接实现

  ---

  ## 10. docs 写作约束（用于新增 / 修改 docs）

  新增或修改 docs 时，优先遵循以下结构：

  # <Feature / Module Name>

  ## Abstract
  - What:
  - When:
  - Not:
  - Key rules:

  ## When to use
  ## When NOT to use
  ## Behavior / Rules
  ## Configuration / API
  ## Examples
  ## Edge Cases / Gotchas
  ## Related Docs

  ### 写作要求
  - 用确定性措辞：必须 / 禁止 / 只会 / 不会
  - 不用模糊词：通常 / 一般 / 可能 / 大多数情况下
  - 一个规则只表达一个行为
  - 必须明确写 Does / Does NOT
  - 有优先级时必须显式编号
  - 示例必须成对：正例 + 反例
  - 不允许“如前文所述”这类隐式依赖
  - 首次出现的概念必须能映射到代码位置、模块名或配置 key

  ---

  ## 11. 完成定义（Final Gate）

  交付前必须自检：

  - 是否先读了 `/docs`
  - 是否只读了最小必要内容
  - 是否根据文档而不是猜测做决策
  - 是否只改了必要代码
  - 是否没有引入多余封装 / 防御 / 回退
  - 是否同步更新了文档
  - 是否给出了验证方式
  - 是否说明了影响、兼容性、风险

  只要任一项未满足，就不能视为完成。
